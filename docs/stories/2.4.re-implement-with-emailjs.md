# Story 2.4 (v3): Re-implement Form Submission with EmailJS

### **Status**: `Done`

### **Story**
**As a** Project Owner, **I want** to implement a client-side form submission using the EmailJS service, **so that** we have full control over the user experience without needing a custom backend.

### **Acceptance Criteria**
1.  The project includes the EmailJS SDK, loaded via a `<script>` tag.
2.  A new JavaScript block or file is created to handle the form's `submit` event.
3.  The script successfully calls the EmailJS API using the correct Service ID, Template ID, and Public Key.
4.  Sensitive keys (like a Private Key, if applicable) are NOT exposed in the client-side code.
5.  A successful submission programmatically redirects the user to `gracias.html`.
6.  The `dev` branch is successfully merged into `main`.
7.  The functionality is verified as working on the live site, `pewma.ai`.

### **Tasks / Subtasks**
* [x] **Task 1**: (Hecho por ti) Crear una cuenta en EmailJS, configurar un servicio de email, y crear una plantilla de correo.
* [x] **Task 2**: (Hecho por ti) Añadir el SDK de EmailJS al `contacto.html`.
* [x] **Task 3**: (Hecho por ti) Escribir el código JavaScript para inicializar EmailJS con tu Public Key y enviar los datos del formulario al `submit`.
* [x] **Task 4**: **Verificar la Seguridad de las Claves.** Confirma que solo estás usando la "Public Key" en tu código JavaScript. La "Private Key" nunca debe estar en el código del frontend.
* [x] **Task 5**: **Fusionar (Merge) la rama `dev` a `main`.**
* [x] **Task 6**: **Verificar en Producción.** Confirmar que el formulario funciona correctamente en `https://pewma.ai`.

### **Dev Notes**
* **Nuevo Stack**: Este cambio introduce una dependencia de JavaScript (EmailJS SDK) al proyecto.
* **Claves de API**: EmailJS utiliza una "Public Key" que es segura de exponer en el lado del cliente. Asegúrate de no haber incluido accidentalmente la "Private Key" o cualquier otra credencial secreta en el código.
